# 稅籍_全國營業登記資料集
## 稅籍資料日批管線（GCIS TaxInfo Daily Pipeline）

本專案是一套 **稅籍資料日批處理管線**，用於每日（或回補）處理政府 GCIS 稅籍 CSV，並將「新增 / 異動」資料**追加寫入主表（歷史保留）**。

核心設計目標：

* **Raw-first**：原始資料一律先完整入庫，作為可稽核依據
* **Legacy-compatible**：沿用既有 `Tmp_TaxInfo` / `TaxInfo` 表結構，不打掉重練
* **Append-only main table**：主表只新增，不覆蓋、不 upsert
* **強制核對**：raw(DATA) vs tmp 筆數不一致，批次直接失敗
* **可日批、可回補、可手動指定 CSV**

## 模組化資料處理與測試管線專案  
## Modular Data Pipeline & Independent Test Framework

本專案由兩個互補但責任清晰的模組組成，目標在於**將資料處理流程從單一腳本，拆解為可組裝、可測試、可擴充的工程化架構**。

專案設計特別著重於「**模組化**」與「**可驗證性**」，以支援後續不同資料來源、不同商業邏輯，以及長期維運與重構需求。

## 專案組成

### 1. `lab_250930v3_模組化版本.py`
**主資料處理模組（Production-oriented Modular Pipeline）**

### 2. `lab_251001_獨立測試功能開發.py`
**獨立測試與驗證模組（Isolated Test & Experiment Framework）**

## 一、`lab_250930v3_模組化版本.py`  
### 主資料處理模組（模組化版本）

### 模組定位
此檔案為**正式資料處理管線的模組化實作版本**，負責將原本集中在單一腳本中的邏輯，拆分為**可重用、可替換的功能單元**。

它的角色相當於：
> 「**生產環境使用的核心處理引擎**」

### 核心設計目標

- **模組化（Modular-first）**  
  每個處理步驟（下載、解析、轉換、寫入、驗證）皆為獨立函式或邏輯區塊，避免單一巨大流程。

- **流程可組裝（Composable Pipeline）**  
  管線不是硬編碼流程，而是由多個模組組合而成，可依需求調整順序或替換實作。

- **資料責任清楚（Clear Responsibility Boundary）**  
  - 資料取得 ≠ 資料轉換 ≠ 資料寫入  
  - 每一層只關心自己的輸入與輸出，不跨層做事。

- **面向正式運行（Production-oriented）**  
  程式結構、錯誤處理與紀錄方式，皆以「長期跑在排程 / 生產環境」為前提設計。

### 設計理念摘要

- 避免「一次寫完所有事情」的腳本風格  
- 為後續資料來源擴充、流程調整預留空間  
- 作為後續所有資料管線的「結構樣板」

## 二、`lab_251001_獨立測試功能開發.py`  
### 獨立測試與驗證模組

### 模組定位
此檔案為**非生產用途的獨立測試與實驗模組**，主要用於：

- 驗證單一功能是否正確
- 嘗試新邏輯、新演算法、新資料處理方式
- 在不影響主流程的情況下進行實驗

它的角色相當於：
> 「**工程師的實驗場與安全沙盒**」

### 核心設計目標

- **測試隔離（Isolation-first）**  
  所有測試邏輯與實驗程式碼，**不直接依賴正式管線流程**，避免影響生產穩定性。

- **快速驗證（Fast Feedback）**  
  可快速針對單一功能進行測試，而不需跑完整批次流程。

- **錯誤容忍（Fail-safe Experiment）**  
  即使測試失敗，也不影響正式模組或資料狀態。

- **支援反覆試驗（Iterative Development）**  
  鼓勵用此檔案進行多次嘗試、修正與比較，而非在主模組中「邊跑邊改」。

### 設計理念摘要

- 將「思考與試驗」從正式流程中拆出  
- 降低實驗成本，避免污染主系統  
- 作為新功能進入主模組前的前哨站

## 三、兩者之間的關係與分工

| 面向 | 模組化版本 | 獨立測試版本 |
| --- | --- | --- |
| 使用場景 | 生產環境 / 排程 | 開發 / 驗證 / 試驗 |
| 穩定性要求 | 高 | 可接受失敗 |
| 程式風格 | 結構化、可維運 | 彈性、可快速修改 |
| 是否影響主流程 | 會 | 不會 |
| 設計目標 | 長期可維護 | 快速學習與驗證 |

## 專案總結

此專案的核心價值不在於「寫了多少功能」，而在於**將資料處理這件事，從腳本升級為工程系統**：

- 主模組負責 **穩定、可重用、可長期運行**
- 測試模組負責 **探索、驗證、降低風險**

這樣的分工，能確保專案在持續擴充的同時，仍然保持結構清晰與風險可控。
